## 1. プロジェクト概要

### 目的
清掃作業員の勤怠管理をデジタル化し、管理業務の効率化を図る。

### 背景
- 現在、LINE・口頭・FAX・カレンダーアプリ・Excelなど複数手段で勤怠や作業実績を管理
- 情報が分散しており、管理者の業務負担が大きい
- 清掃業は利益率が低く、業務効率化が重要

### ユーザー
| ロール | 人数 | 説明 |
|--------|------|------|
| 作業員（Worker） | 8名 | 清掃作業を行う。日本人2名、スリランカ人6名 |
| 管理者（Manager） | 2名 | 勤怠管理・現場割り振り・評価を行う |

### 特記事項
- 日本語を読めない作業員が多い（多言語対応が必須）
- スマートフォン未所持の作業員がいる（共有端末＋アカウント制）
- 日本語を話せるスリランカ人リーダー（マリク）が在籍
- 作業指示書（FAX）は日本語のままでも問題ない（作業員は教育済み）
- 1日あたり作業員1人が担当する現場数は 1〜5件

## 2. 業務フロー概要（2026-07-15 MTG で確認）

### 1日の流れ

```
事務所に集合 → 管理者が作業指示（口頭/LINE）
    → 車（9割）または電車（1割）で現場へ移動
    → 2〜4チームが並行して各現場で作業
    → 作業完了後、管理者へ完了連絡
    → 基本は直帰（事務所に戻る場合もあり）
```

### 現場の特徴
- 現場の鍵：9割が現場設置（ダイヤル式/暗証番号）、1割は管理者が事前に受け取り渡す
- 現場住所はGoogleマップで作業員へ共有
- 同じ建物でも部屋番号が違えば別の案件として扱う
- リピート案件（同じ物件への再依頼）がある
- 顧客都合で当日急に別現場に変更されることもある

### 給与関連
- 給与計算は毎月10日締め、Excel運用を継続
- 給与明細はExcel→PDF化してLINEで送付（→アプリ化で省力化を目指す）
- 手当・交通費はアプリ管理しない
- 残業は30分単位
- 休憩時間の打刻は不要。管理者が確定勤務時間で調整する

## 3. アーキテクチャ方針

### アプリ構成
- 1つのWebアプリケーション
- ログイン後、ロール（worker / admin）で画面を切替
- モバイルファースト（作業員はスマホで使用）

### 技術スタック
- フロントエンド: HTML + CSS + JavaScript（Vanilla）
- データ保存: localStorage（将来的にバックエンド追加予定）
- 対応言語: 日本語 / シンハラ語 / 英語
- 翻訳API: Google Cloud Translation（自由入力テキストの翻訳用）

### フォルダ構成

```
kintai-app/
├── docs/                    ← 設計ドキュメント群（現在のフェーズ）
│   ├── CONTEXT.md           ← このファイル
│   ├── domain/              ← DDDドキュメント
│   ├── design/              ← 画面設計・遷移図
│   ├── specs/               ← 機能仕様書（SPEC-xxx.md）
│   └── tasks/               ← 保留事項・タスク管理
├── src/                     ← ソースコード（実装フェーズで作成）
│   ├── index.html
│   ├── css/
│   │   └── style.css
│   ├── js/
│   │   ├── app.js           ← エントリポイント
│   │   ├── auth/            ← 認証コンテキスト
│   │   ├── attendance/      ← 勤怠管理コンテキスト
│   │   ├── site-mgmt/       ← 現場管理コンテキスト
│   │   ├── i18n/            ← 多言語対応コンテキスト
│   │   ├── ui/              ← 共通UIコンポーネント
│   │   └── utils/           ← ユーティリティ関数
│   └── assets/              ← 画像・アイコン等
└── learning/                ← 学習資料（開発とは独立）
```

> コンテキストマップの各コンテキスト = `src/js/` 配下のディレクトリに対応。

### データストア方針

| Phase | 方式 | 説明 |
|-------|------|------|
| Phase 1-2 | localStorage | フロントエンドのみ。プロトタイプとして十分 |
| Phase 3以降 | 検討中 | Firebase / Supabase / 自前API等。ユーザー数や要件に応じて選定 |

- localStorage使用時も、データアクセスは**リポジトリパターン**で抽象化する
- 将来のバックエンド移行時はリポジトリの実装を差し替えるだけで対応

```javascript

const DailyRecordRepository = {
  save(record) { /* localStorage.setItem(...) */ },
  findByDate(workerId, date) { /* localStorage.getItem(...) */ },
  findByMonth(workerId, yearMonth) { /* ... */ }
};
```

### 多言語対応の方針

| 対象 | 方法 | 説明 |
|------|------|------|
| UI表示（ボタン名、ラベル等） | i18n 翻訳ファイル（JSON） | 事前に用意した翻訳を表示。APIは不要 |
| 自由入力テキスト（備考欄等） | 翻訳API（Google Cloud Translation） | 送信時にAPIで自動翻訳。原文も保持する |
| 作業指示書（FAX） | 翻訳不要 | 日本語のまま。作業員は教育済み |

- UIの表示言語はアカウントの `language` 設定に従う
- 作業員（worker）: シンハラ語または英語で表示
- 管理者（admin）: 日本語で表示

### データモデルの基本方針
- **物件（Property）** と **案件（Job）** を分離して管理
  - 物件: 住所、棟名、部屋番号、リピートフラグ → 継続的に保存
  - 案件: 作業予定日、担当者、鍵情報、作業指示書画像 → 作業日ごとに発生
- 詳細は `docs/domain/domain-model.md` を参照

### パスワード管理方針
- パスワードの管理は管理者が一括で行う
- 作業員にはパスワード変更機能を提供しない
- 打刻修正は管理者のみが行える。作業員は修正不可
- セキュリティより工程削減を重視

### 制約
- フレームワークは使用しない（Vanilla JS）
- オフラインでも基本機能（打刻）が使えること
- レスポンシブデザイン（スマホ・タブレット・PC対応）

### 作業員向け表示ルール（重要）
- **作業員向け画面（W-01等）には出退勤時刻・勤務時間を表示しない**
- 出勤/退勤ボタンはステータス変更のみ（内部的に時刻は記録する）
- 給与に関わる時間管理はすべて管理者がA-01ダッシュボードで行う
- 理由: 細かい時刻を見せると給与計算とのズレで不信を招くため
- 参照: 境界コンテキスト「勤怠管理」ルール#15

## 5. 開発Phase

| Phase | 範囲 | SPEC | 目的 |
|-------|------|------|------|
| **Phase 1（MVP）** | ログイン、出退勤打刻、現場作業管理、管理ダッシュボード、勤務実績 | SPEC-001〜005 | 最小限の打刻ループが回る状態 |
| **Phase 2** | 作業報告、仮登録 | SPEC-006〜007 | 現場作業の報告機能 |
| **Phase 3** | 物件/案件管理、割り振り、アカウント管理、作業員詳細 | SPEC-008〜011 | 管理者向け機能の拡充 |
| **Phase 4** | 作業評価、多言語対応、勤務履歴/給与明細、設定 | SPEC-012〜015 | 仕上げ・横断機能 |

```
  Phase 1: ログイン → 出退勤 → 現場作業 → 管理者確認・確定 → 勤務実績
  Phase 2: 作業報告 → 仮登録
  Phase 3: 物件/案件管理 → 割り振り → アカウント → 作業員詳細
  Phase 4: 評価 → 多言語 → 履歴/給与 → 設定
```

## 6. 開発フロー

1. **DDD** でドメインモデルを定義（`docs/domain/`） ← 完了
2. **SDD** で機能仕様を作成（`docs/specs/`） ← **現在ここ**
3. **AIエージェント**（Antigravity / Claude Code）が仕様に基づき実装
4. **レビュー** — 仕様と照合して検証
5. **Git** — ブランチ運用でチーム開発

### テスト方針

| レベル | 方法 | 対象 |
|--------|------|------|
| **手動テスト** | ブラウザで操作して確認 | UI表示、画面遷移、レスポンシブ |
| **受入テスト** | SPECの受入基準チェックリスト | 各機能の完成判定 |
| **ユニットテスト**（任意） | テストフレームワーク未定 | 勤務時間計算等の重要ロジック |
